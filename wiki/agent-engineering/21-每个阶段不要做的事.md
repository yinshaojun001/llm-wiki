# 每个阶段不要做的事

> 知道不该做什么和知道该做什么一样重要。

---

## 原理筑基阶段（第 1-12 周）

| 不要做 | 为什么 | 替代做法 |
|---|---|---|
| 读《Attention Is All You Need》原论文 | 写给研究者的，不是给你入门的 | 看 3Blue1Brown 动画 + Jay Alammar 图解 |
| 手推 softmax 梯度 | Agent 工程师不写反向传播 | 理解"softmax 把数值变成概率分布"即可 |
| 从零实现完整 Transformer | 第一周就做会劝退 | 用 nanoGPT 读 50 行 CausalSelfAttention |
| 纠结"头数为什么是 8 不是 16" | 炼丹师的事 | 理解"多头 = 多维度关注"即可 |
| 花大量时间调 Prompt 玩 | 没有评估的调优是玄学 | 先建 Golden Test Set 再调 |
| 学各种 Prompt 模板库 | 模板会过时 | 学 CoT/Few-shot 原理再自己写 |
| 纠结 Embedding 模型训练原理 | 你是用 Embedding，不训练它 | 调 `text-embedding-3-small` API 观察效果 |
| 过早学 Milvus/Pinecone | 入门太重 | Chroma 本地跑通再升级 |
| 追求"完美分块策略" | 没有银弹，取决于文档类型 | 256/512/1024 三分法各试一遍，选效果最好的 |
| 直接上 LangChain 全套 | 封装太厚不理解本质 | 先手写 ReAct 循环，再用框架 |

---

## 组件深入阶段（第 13-24 周）

| 不要做 | 为什么 | 替代做法 |
|---|---|---|
| 试图让 Agent 记住一切 | 记忆管理的核心是选择性遗忘 | 实现衰减机制，按重要性/时效性排序 |
| 所有任务都用 Plan-and-Solve | 简单任务过度规划浪费 Token | 简单任务用 ReAct，复杂多步任务用 P&S |
| 认为 Planning 越复杂越好 | 实用 > 炫技 | 以"完成任务的步骤数 × Token 消耗"作为指标 |
| 工具越多越好 | 工具列表爆炸 → LLM 选错率上升 | 5-8 个核心工具，按需加载 |
| 简单任务用多 Agent | Token 翻 3 倍效果没提升 | 单 Agent 能解决的绝不用多 Agent |
| 让子 Agent 有不受限的 session 权限 | 安全灾难 | 默认最小权限，嵌套深度 ≤ 2 |
| 认为"感觉 Agent 变差了"就是变差了 | 没有量化就没有优化 | 每次变更跑 Golden Test Set 对比 |

## 应用开发阶段（第 25-36 周）

| 不要做 | 为什么 | 替代做法 |
|---|---|---|
| 用 vibe check 代替量化评估 | "感觉还行"不能指导改进 | Golden Test Set + LLM-as-Judge 自动化 |
| 跳过可观测性 | 出问题不知道哪里坏了 | 每个 Agent 项目接入 Langfuse |
| 只测 happy path | 生产环境 80% 的问题是边界情况 | Golden Test Set 必须包含边界和容易出错的 case |
| 一个 Agent 项目做 3 个月 | 做多比做深更有价值 | 3 个不同类型的 Agent，积累广度 |
| 代码不开源 | 面试官看不到你的代码 | GitHub public repo + 好 README |

---

## 落地生产化阶段（第 37-52 周）

| 不要做 | 为什么 | 替代做法 |
|---|---|---|
| 一次性全量上线 | Agent 的失败模式不可预测 | Shadow → Canary → 5% → 25% → 全量 |
| 跳过审计日志 | 合规必备，出问题没得查 | 每个 Agent 操作都记日志 |
| 绑死一个模型供应商 | 模型可能被弃用 | 多模型路由架构 |
| 过度设计 | 先跑通再优化 | MVP → 数据验证 → 迭代优化 |
| 只看 Token 成本不看业务价值 | 省了 30% 成本但慢了 5 倍不值得 | 成本优化后必须跑同一套评估对比 |
| 不写技术博客 | 简历上一个 link 胜过十行描述 | 每月一篇，写你最有心得的一个 topic |
