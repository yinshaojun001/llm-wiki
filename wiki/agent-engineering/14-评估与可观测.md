# 评估与可观测性

> 这是从 Demo 到生产的门槛。没有评估的 Agent 是在飞盲。

---

## 一、评估体系

### 三层评估

```
L1 单元评估  → 检索质量（Recall/Precision/MRR）、工具选择准确率
L2 行为评估  → 推理路径是否正确、工具调用顺序是否合理
L3 端到端    → 目标成功率、用户满意度
```

### Golden Test Set

- 30-200 个人工标注的典型用例
- 覆盖：正常路径 + 边界情况 + 已知容易出错的 case
- 每次改 Prompt、换模型、改分块策略后，自动跑全量
- **不通过不发版**（和 CI/CD test stage 一样）

### LLM-as-a-Judge

用另一个 LLM 给 Agent 的回答打分：

| 维度 | 衡量什么 |
|---|---|
| 正确性 | 答案对吗 |
| 完整性 | 该说的都说了吗 |
| 忠实性 | 有没有瞎编（幻觉） |
| 工具选择 | 该用的工具都用对了吗 |

> 局限性：Judge 也会有偏见，需要人工抽样校准。

### Ragas 评估框架（RAG 专项）

| 指标 | 衡量什么 |
|---|---|
| Faithfulness | 回答是否忠于检索到的原文 |
| Answer Relevancy | 回答是否切题 |
| Context Precision | 检索到的文档是否精准 |
| Context Recall | 该检索到的文档是否都检索到了 |

## 二、可观测性

### 全链路 Tracing

每个 LLM 调用、工具执行、检索、Agent 间通信，都打上 Trace。

```
用户请求
  └─ Agent 推理（Span 1） → LLM 调用（Span 1.1）→ Token 消耗
  └─ 工具调用（Span 2）   → get_weather(SF) → 耗时 230ms
  └─ Agent 推理（Span 3） → LLM 调用（Span 3.1）
  └─ 最终输出（Span 4）
```

### 关键指标

| 指标 | 为什么重要 | 告警阈值（参考） |
|---|---|---|
| 延迟 p50 / p95 | 用户体验 | p95 > 5s 告警 |
| Token 消耗 / 任务 | 成本控制 | 超过预算 2× 告警 |
| 工具调用成功率 | Agent 可靠性 | < 95% 告警 |
| 目标成功率 | 业务 KPI | < 80% 告警 |
| 幻觉率 | 信任 | > 5% 告警 |
| 死循环检测 | 防止浪费 | 同一工具连续调 5 次告警 |

### 工具选型

| 工具 | 定位 |
|---|---|
| **Langfuse** | 开源，Agent 专用，Trace + 评估 + 成本追踪 |
| **Arize** | 企业级，PagerDuty 在用 |
| **LangSmith** | LangChain 生态原生 |
| **OpenTelemetry** | 通用标准，作为 Trace 传输层 |

## 三、评估驱动开发（ADD — Agent-Driven Development）

```
1. 写 Golden Test Set（定义"好"是什么）
2. 写 Prompt / 搭 Agent
3. 跑评估 → 看分数
4. 分析失败 case → 改 Prompt / 改检索 / 换模型
5. 再跑评估 → 确认没有回退
6. 把新发现的失败 case 加入 Golden Test Set
```

和 TDD 一样的逻辑——先定义期望结果，再开发，最后验证。
