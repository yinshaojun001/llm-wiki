# 第一阶段：原理筑基

> 第 1-12 周 | 目标：理解 LLM 底层运行机制，能调用 API，能写高质量 Prompt，能搭 RAG。

---

## 第 1-2 周：Transformer 原理

**学什么**：文本进去 → token 化 → 向量 → 多层 Attention → 输出，每一步发生了什么。

### 怎么学

| 天   | 内容         | 方法                                                                                                   |
| --- | ---------- | ---------------------------------------------------------------------------------------------------- |
| 1-2 | 先看动画，别碰论文  | 3Blue1Brown "Attention in transformers"（B站/YouTube，26min）→ Jay Alammar "The Illustrated Transformer" |
| 3-4 | 跑代码，别手写    | `pip install transformers`，加载 gpt2，观察 tokenization，跑一个 forward pass                                  |
| 5   | nanoGPT 实验 | 读 Karpathy nanoGPT `model.py` 的 `CausalSelfAttention` 类（~50 行），理解 `q @ k.transpose`                  |
| 6-7 | Java 类比消化  | 对着 [[13-Java工程师视角]] 里的类比表，用自己的话写一遍                                                                   |

### 产出

一个 Jupyter Notebook，加载 gpt2 完成分词+生成，每步有注释。

### 过关标准

- 一段文本进 Transformer 经过了哪些步骤？（口头说清）
- 为什么 GPT 是逐 token 生成？
- Context Window 从 4K 涨到 128K 的代价是什么？

### 一定不要

- ❌ 读原论文
- ❌ 手推梯度
- ❌ 从零实现 Transformer
- ❌ 纠结"头数为什么是 8 不是 16"

## 第 3-4 周：LLM API 与 Prompt Engineering

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | API 基础 | 注册 OpenAI/Anthropic 账号，跑通第一个 API 调用。理解 `model`、`temperature`、`max_tokens` |
| 3-4 | System Prompt 设计 | 设定角色、行为边界、输出格式。实验：同一个问题不同 system prompt，对比输出 |
| 5 | Chain-of-Thought | Prompt 里加"Let's think step by step"，观察推理质量变化 |
| 6-7 | Few-shot + 结构化输出 | 2-3 个示例让模型模仿格式；JSON Mode 返回可解析结果 |

### Java 类比

- System Prompt ≈ 接口规范/JavaDoc
- Few-shot ≈ 单元测试的 Given-When-Then
- Temperature ≈ 0 = 严格按规范，1 = 允许变体

### 产出

一个命令行聊天工具（Python ~100 行），支持 System Prompt 配置、对话历史、JSON 输出。

### 过关标准

- 写出能让模型"只返回 JSON，不废话"的 system prompt
- 用 2-3 个 few-shot 让模型完成特定格式输出
- 知道 temperature=0 和 1 的适用场景

### 一定不要

- ❌ 花大量时间调 Prompt 玩（没评估的调优是玄学）
- ❌ 学 Prompt 模板库（模板过时快，理解为什么有效才持久）

## 第 5-6 周：Embedding 与向量检索

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | Embedding 原理 | 用 OpenAI `text-embedding-3-small` 把文本变向量，观察"相似句子→相似向量" |
| 3-4 | Chroma 入门 | `pip install chromadb`，100 篇文档 embedding 后存进去，自然语言搜索 |
| 5-6 | 语义相似度 | 余弦相似度计算、Top-K 检索、PCA 降维可视化 |
| 7 | 理解局限 | 故意搜会被关键词误导的查询，体会"关键词搜索"和"语义搜索"的差异 |

### Java 类比

- Embedding ≈ 把对象序列化成 `double[768]`，相似对象数组值接近
- 向量检索 ≈ `Stream.sorted(Comparator.comparing(Obj::similarity)).limit(K)`
- ANN ≈ HashMap 的 O(1) 近似版，用精度换速度

### 产出

命令行工具：对任意文件夹里的 txt/md 建索引，自然语言搜索返回 Top-5 + 相似度分数。

### 过关标准

- 解释"向量检索"和"数据库 LIKE 查询"的本质区别
- 说出 ANN 和精确 KNN 的取舍

### 一定不要

- ❌ 纠结 Embedding 模型的训练原理
- ❌ 过早学 Milvus/Pinecone

## 第 7-8 周：RAG 基础

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | Naive RAG | 用户提问 → embedding → 检索 Top-K → 拼到 Prompt → LLM 回答。跑通最小闭环 |
| 3-4 | 分块策略 | 同一份 PDF，128/256/512/1024 token 不同分块对比检索质量 |
| 5-6 | 混合检索 + Reranking | BM25 关键词 + 向量语义检索 → 结果融合 → Reranker 精排 |
| 7 | 引用溯源 | 让 LLM 注明信息来源（哪个文档的哪一段），不瞎编 |

### Java 类比

- RAG ≈ 给新同事代码库索引让他搜了再回答
- Chunking ≈ 数据库分页——太大一次查太多，太小需要多次 round-trip
- Reranking ≈ `ORDER BY relevance DESC` 但 relevance 由专门模型计算

### 产出

一个能回答 PDF 内容的 RAG 应用（命令行或 Gradio 界面），每句回答带引用来源。

### 过关标准

- RAG 和"直接把文档塞进 Context Window"的区别
- 什么时候用 RAG，什么时候用长上下文模型
- 三种分块策略的适用场景

### 一定不要

- ❌ 追求"完美分块"（没有银弹）
- ❌ 把 RAG 当万能药（少量文档直接放 context window 更简单）

## 第 9-10 周：Function Calling

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | 第一个 Tool | 定义 `get_weather(city)`，让 LLM 在用户问天气时自动调用 |
| 3-4 | 多工具选择 | 同时给 5 个工具（天气、搜索、计算器、时间、汇率），看 LLM 怎么选 |
| 5-6 | Schema 设计实验 | 同一功能写三版 tool description，对比 LLM 的选择准确率 |
| 7 | 错误处理 | 工具调用失败时 LLM 怎么恢复——重试、换工具、告诉用户 |

### Java 类比

- Function Calling ≈ 接口定义——方法签名 + 描述，LLM 决定何时调用哪个
- Tool Schema ≈ REST API 文档——写得好调用方自然用对
- 错误恢复 ≈ try-catch + fallback

### 产出

一个带 5+ 工具的 Agent，能查天气、搜索网页、计算、查时间、换算汇率。

### 过关标准

- Tool description 怎么写能让 LLM 准确选择
- 完整 tool calling 流程：LLM 输出 tool_call → 你的代码执行 → 结果返回 LLM → LLM 继续推理
- 处理过工具调用失败

### 一定不要

- ❌ 认为工具越多越好（工具越多，选错概率越大）

## 第 11-12 周：Agent 基础范式 — ReAct

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-3 | 手写 ReAct 循环 | 不用任何框架：Thought → Action → Observation → Thought... 直到 Final Answer |
| 4-5 | 用框架实现 | 同一个任务用 OpenAI Agents SDK 实现，对比手写版 |
| 6-7 | 综合 Demo | "查一下旧金山今天天气，换算成摄氏度，和北京对比"——需多步推理+多工具调用 |

### Java 类比

- ReAct 循环 ≈ `while (!taskComplete) { think(); act(); observe(); }`
- Agent 状态 ≈ 状态机——每个 Observation 决定下一个 Action

### 产出

一个完整的 ReAct Agent（命令行），能处理多步推理+多工具调用任务。

### 过关标准

- 画出 ReAct 循环的完整流程图
- 知道 Agent 在哪步最容易出问题（工具调用失败）
- 手写版和框架版都写过

### 一定不要

- ❌ 上来就用 LangChain 全套（先手写理解本质）
