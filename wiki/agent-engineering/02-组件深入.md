# 第二阶段：组件深入

> 第 13-24 周 | 目标：掌握 Memory、Planning、Tool Use、Multi-Agent、Safety 五大组件的实现。

---

## 第 13-14 周：Memory 系统

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | 短期记忆 | Context Window 管理——对话历史太长怎么办？滚动窗口 vs 摘要压缩 |
| 3-4 | 工作记忆 | Agent 边做边记中间结果——scratchpad 模式 |
| 5-6 | 长期记忆 | 用 Chroma 存用户偏好和知识，下次对话自动检索相关记忆 |
| 7 | 记忆管理 | 实现简单"遗忘"逻辑（时间衰减 + 重要性保留），不是无限堆积 |

### Java 类比

- 短期记忆 ≈ 本地变量（出作用域就没了）
- 工作记忆 ≈ ThreadLocal（当前任务独享）
- 长期记忆 ≈ 数据库（持久化，按需查）
- 记忆衰减 ≈ LRU Cache 淘汰策略

### 产出

一个带三层记忆的 Agent——记得当前对话、当前任务状态、历史偏好。

### 过关标准

- 画出三层记忆数据流图
- 知道什么时候该压缩、什么时候该检索、什么时候该遗忘
- 理解 Session 外部化（append-only event log）和 context window 管理是两回事

### 一定不要

- ❌ 试图记住一切——记忆管理的核心是选择性遗忘

> 详见 [[11-Agent核心组件]] Memory 节、[[concepts/agent-memory]]、[[concepts/agent-session]]

## 第 15-16 周：Planning 策略

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | ReAct vs Plan-and-Solve | 同一个任务两种策略，对比步骤数、Token 消耗、准确率 |
| 3-4 | Tree-of-Thoughts | 每步探索 2-3 种方案，理解"搜索空间"概念 |
| 5-6 | Reflexion | Agent 执行出错后自我反思——"我刚才哪里做错了？下次怎么避免？" |
| 7 | 策略对比 | 四种策略跑同一个复杂任务，写对比报告 |

### Java 类比

- ReAct ≈ 敏捷迭代（边做边调整）
- Plan-and-Solve ≈ 瀑布模型（先设计再编码）
- Tree-of-Thoughts ≈ 每个技术决策做 POC 选最优方案
- Reflexion ≈ 事故复盘 postmortem

### 产出

一个能跑 ReAct / P&S / Reflexion 三种策略的实验框架 + 对比报告。

### 过关标准

- 什么时候用 ReAct（简单任务），什么时候用 Plan-and-Solve（多步任务）
- Reflexion 和普通 retry 的区别（Reflexion 改变策略，retry 只是重复）
- Planning 的主要代价（Token 消耗、延迟增加）

### 一定不要

- ❌ 所有任务都用 Plan-and-Solve（简单任务过度规划浪费 Token）
- ❌ 认为 Planning 越复杂越好——实用 > 炫技

> 详见 [[11-Agent核心组件]] Planning 节

## 第 17-18 周：Tool Use 工程

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | Schema 设计规范 | 研究 OpenAI/Anthropic 官方工具示例，总结"好 schema"的共性 |
| 3-4 | 风险分级 | 只读 → 读取+过滤 → 写入 → 高风险（发邮件/部署/支付） |
| 5-6 | MCP 协议 | 搭建一个 MCP Server，让 Agent 通过 MCP 调用工具 |
| 7 | 沙箱执行 | Docker 沙箱内执行 Agent 生成的代码，限制网络/文件系统/进程 |

### Java 类比

- 风险分级 ≈ 权限系统（读/写/管理员，每层有审批）
- MCP ≈ Service Mesh（服务间通信标准化）
- 沙箱 ≈ JVM SecurityManager（限制代码执行边界）

### 产出

一个 8+ 工具的工具箱，含风险分级、MCP Server 接入、沙箱执行。

### 过关标准

- 写出"让 LLM 高概率选对工具"的 schema
- 理解工具安全分级模型
- 跑通过 MCP Server + Agent 端到端调用

> 详见 [[11-Agent核心组件]] Tool Use 节、[[concepts/mcp]]、[[concepts/skills-architecture]]

## 第 19-20 周：多 Agent 协作

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | Orchestrator 模式 | 一个主 Agent 分派任务给 2-3 个子 Agent，收集结果，合并回答 |
| 3-4 | Handoff 模式 | Agent A 完成一部分，移交上下文给 Agent B 继续 |
| 5-6 | Debate 模式 | 两个 Agent 各出方案，第三个 Agent 评判选出更好的 |
| 7 | 完整 Demo | 研究员收集信息 → 分析师处理数据 → 写手出报告 |

### Java 类比

- Orchestrator ≈ API Gateway（路由分发，聚合响应）
- Handoff ≈ 责任链模式
- Debate ≈ 多人 Code Review

### 产出

一个 3 Agent 协作的 Demo（研究员→分析师→写手）。

### 过关标准

- 什么时候用单 Agent，什么时候需要多 Agent
- 多 Agent 的核心成本（Token 翻倍、协调开销、结果不一致）
- 能设计 Agent 间的通信协议

### 一定不要

- ❌ 简单任务用多 Agent（Token 翻 3 倍效果没提升）
- ❌ 让子 Agent 有不受限的 session 管理权限

> 详见 [[11-Agent核心组件]] Multi-Agent 节

## 第 23-24 周：安全与护栏

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | Input Guardrail | 实现 Prompt 注入检测——用户输入藏"忽略之前指令"，Agent 能被骗吗？ |
| 3-4 | Output Guardrail | LLM 输出后内容审核——敏感信息、幻觉检测、格式校验 |
| 5-6 | Human-in-the-Loop | 高危操作（删文件、发邮件）需人工确认后才执行 |
| 7 | 循环/超限检测 | Agent 死循环（反复调同一工具）→ 自动熔断 |

### Java 类比

- Input Guardrail ≈ XSS/SQL 注入防护
- Output Guardrail ≈ API Response Validation
- Human-in-the-Loop ≈ 审批流程
- 熔断 ≈ Hystrix/Sentinel

### 产出

一个带四层防护（输入→执行→输出→熔断）的 Agent 框架。

### 过关标准

- 说出 Agent 安全的四道防线
- 跑通过一个 Prompt 注入攻击并成功拦截
- 理解 Human-in-the-Loop 的工程开销和不可替代的价值

> 详见 [[11-Agent核心组件]] Safety 节
