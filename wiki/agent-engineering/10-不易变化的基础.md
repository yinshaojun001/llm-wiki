# 不易变化的基础知识

> 这些是无论上层框架如何更迭都会用到的核心原理，优先级最高。学机制，不学名字。

---

## 一、Transformer 与 Attention

### Self-Attention

每个词根据与所有其他词的相关性，动态重新加权自己的表示。

```
Attention(Q,K,V) = softmax(QKᵀ/√dₖ) × V
```

### 关键概念

| 概念 | 说明 | Java 类比 |
|---|---|---|
| Self-Attention | 每个词动态关注所有其他词 | 动态加权路由：Query 匹配 Key，聚合 Value |
| Multi-Head | 多个并行关注通道 | 多线程同时检查代码的不同方面 |
| Residual Connection | 原始信息总能绕过中间层 | try-catch 的 finally 块 |
| Layer Normalization | 归一化层输出 | 数据标准化到同一尺度 |
| Positional Encoding | 注入位置信息 | 数组索引 |
| KV Cache | 缓存已算过的 Key/Value | Redis 缓存，空间换时间 |

### Decoder-Only 架构

GPT 系模型的范式：自回归生成，逐 token 出字。每个新 token 能看到之前的所有 token（因果注意力）。

### Context Window 退化

长上下文会出现"IQ 衰减"——窗口越长，注意力被稀释。128K 窗口实际可靠的可能只有 32K。根本原因：注意力被过多 token 稀释，中间位置的 token 容易被"遗忘"。

> 学习路径：3Blue1Brown 动画 → Jay Alammar 图解 → Karpathy nanoGPT 代码

## 二、Prompt Engineering

### 核心范式

| 技术 | 说明 | 何时用 |
|---|---|---|
| System Prompt | 设定角色、行为边界、输出格式 | 每次调用都带 |
| Chain-of-Thought | 让模型展示推理步骤 | 复杂推理任务 |
| Few-shot | 给 2-3 个示例让模型模仿 | 需要特定输出格式时 |
| 结构化输出 | JSON Mode / Function Calling | 需要机器可解析结果 |

### 本质理解

- In-Context Learning：为什么给例子模型就能照做——Attention 在示例和目标间建立了关联
- CoT 有效的原因：展示推理步骤限制了模型的"跳跃"空间，强迫它按逻辑推
- Few-shot 的局限：示例太多反而稀释注意力，2-3 个最佳

> 不是记住 Prompt 模板，是理解为什么这个 Prompt 有效。

## 三、RAG（检索增强生成）

### 核心流程

```
文档 → 分块 → Embedding → 向量数据库
用户提问 → Embedding → 检索 Top-K → 拼到 Prompt → LLM 回答
```

### 三个关键决策

| 决策 | 选项 | 权衡 |
|---|---|---|
| 怎么分块 | 固定大小 / 语义 / 递归 | 太大稀释精度，太小丢失上下文 |
| 怎么检索 | 关键词 / 向量 / 混合 | 关键词精准但漏同义词，向量语义好但会漏精确匹配 |
| 怎么精排 | 无 / Cross-encoder Reranker | 精排提升质量但增加延迟 |

### 检索质量

- **Recall**：该找的都找到了吗
- **Precision**：找到的都相关吗
- **MRR**：第一个相关结果排在第几位

## 四、Function Calling / Tool Use

### 协议

OpenAI Function Calling 格式是事实标准：

```json
{
  "name": "get_weather",
  "description": "Get current weather for a city",
  "parameters": {
    "city": {"type": "string", "description": "City name"}
  }
}
```

### 关键认知

- Schema 质量决定选择准确率——描述写得不好，Agent 就不知道该什么时候用什么工具
- 工具不是越多越好——工具越多，LLM 选错概率越大
- MCP（Model Context Protocol）是工具接入的标准化协议，正成为行业共识

## 五、Agent 评估

### 核心指标

| 指标 | 衡量什么 |
|---|---|
| 目标成功率 | Agent 有没有完成用户要的事 |
| 工具选择准确率 | Agent 选对工具了吗 |
| 幻觉率 | Agent 瞎编了吗 |
| 延迟 p95 | 95% 的请求在多久内完成 |
| 单任务成本 | Token 消耗 × 单价 |

### 评估方法

- **Golden Test Set**：30-200 个人工标注的典型用例，每次变更都跑
- **LLM-as-a-Judge**：用另一个 LLM 打分（正确性、完整性、是否有幻觉）
- **行为测试**：不是测输出文本，是测推理路径和工具选择
- **持续回归**：脚本化，每次改 Prompt/换模型/改分块策略后自动出对比报告
