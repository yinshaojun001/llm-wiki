# 第三阶段：应用开发

> 第 25-36 周 | 目标：能独立交付生产可用的 Agent 应用。每月一个完整项目。

---

## 第 25-26 周：Agent 评估体系

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | Golden Test Set | 手工创建 20-30 个典型用例，覆盖正常路径 + 边界 + 易错 case |
| 3-4 | LLM-as-a-Judge | 用另一个 LLM 给 Agent 回答打分——正确性、完整性、是否有幻觉 |
| 5-6 | Ragas 评估框架 | 量化 RAG 质量——Faithfulness、Answer Relevancy、Context Precision |
| 7 | 持续回归 | 脚本化——每次改 Prompt/换模型/改分块策略后，自动跑全量评估出对比报告 |

### Java 类比

- Golden Test Set ≈ 回归测试套件——每次提交跑一遍
- LLM-as-a-Judge ≈ Code Review
- 评估 Pipeline ≈ CI/CD test stage——不通过不发版

### 产出

一套自动化评估 Pipeline，30+ 用例，每次变更一键出报告。

### 过关标准

- 能用 3 个以上维度评估 Agent 输出
- 理解 LLM-as-a-Judge 的局限性（Judge 也会有偏见）
- 评估报告能说清楚"这次变更变好了还是变差了"

### 一定不要

- ❌ 用 vibe check（"感觉还行"）代替量化评估
- ❌ 做太多评估维度（>5 个会疲劳，聚焦业务相关指标）

## 第 27-28 周：可观测性

### 怎么学

| 天 | 内容 | 方法 |
|---|---|---|
| 1-2 | OpenTelemetry 基础 | 理解 Trace/Span/Attribute——给 Agent 每一步打追踪标签 |
| 3-4 | Langfuse 接入 | 把 Agent 所有 LLM 调用、工具调用、检索结果接入 Langfuse 面板 |
| 5-6 | 成本追踪 | 按任务/用户/session 拆分 Token 消耗，计算单次任务成本 |
| 7 | 告警规则 | 延迟超阈值、Token 异常飙升、工具调用失败率上升 → 自动告警 |

### Java 类比

- Trace ≈ 分布式链路追踪（一个请求经过多个微服务的完整路径）
- Span ≈ 链路中的一个步骤
- Langfuse ≈ Grafana + Prometheus——仪表盘 + 监控 + 告警

### 产出

全链路追踪面板，能看到每次 Agent 调用的完整轨迹 + Token 消耗 + 工具调用链。

### 过关标准

- 能根据 Trace 定位 Agent 在哪一步出错
- 能算出单次任务的成本
- 知道 p50 和 p95 延迟的区别和各自意义

> 详见 [[14-评估与可观测]] 第二节

## 第 29-30 周：项目 1 — 个人效率 Agent

### 做什么

集成日历 + 邮件 + 简单记忆的个人 AI 助理。

### 关键功能

- 自然语言查询"下周二的会议"
- 自动发邮件、读新邮件摘要
- 记住你的偏好（"以后所有会议提醒提前 10 分钟"）

### 技术栈

OpenAI Agents SDK + Gmail API + Google Calendar API + Chroma 记忆

### 产出

可用的个人助理（命令行即可，不需要 UI）。

---

## 第 31-32 周：项目 2 — 垂直领域 RAG Agent

### 做什么

面向特定行业的文档问答 Agent（选你熟悉的领域：客服/法律/技术文档）。

### 关键功能

- 上传行业文档建知识库
- 自然语言提问，Agent 搜文档回答 + 引用原文出处
- 处理"我不确定"——找不到相关信息时诚实说不知道，不瞎编

### 技术栈

LlamaIndex/LangChain + Chroma/Milvus + Reranker + 引用溯源

### 产出

可用的领域问答 Agent + 评估报告（Faithfulness / Answer Relevancy）。

---

## 第 33-34 周：项目 3 — 代码 Agent

### 做什么

一个能读代码库、定位 Bug、提议修复的 Agent。

### 关键功能

- 理解项目结构（读目录树和关键文件）
- 根据 Bug 描述定位相关代码
- 生成修复方案（diff 格式）
- （可选）Docker 沙箱里执行测试验证修复

### 技术栈

LangGraph + Docker SDK + GitPython

### 产出

可用的代码助手 + Demo 视频（修一个真实 Bug）。

---

## 第 35-36 周：项目 4 — 复杂工作流 Agent

### 做什么

10+ 步骤、有分支和条件判断的业务流程 Agent。选你熟悉的业务场景（如电商订单处理：接单→验库存→计算运费→生成发票→通知仓库→追踪物流）。

### 关键功能

- 条件分支（有库存/无库存处理逻辑不同）
- 断点恢复（执行到第 7 步挂了，重启从第 7 步继续）
- 人在回路（超预算订单需人工审核）
- 并行执行（同时查库存和算运费）

### 技术栈

LangGraph（状态机 + Checkpoint）+ FastAPI + Redis

### 产出

生产可用的工作流 Agent + 系统架构图 + 评估报告。

---

## 四个项目的共同要求

- [ ] 每个项目有 Golden Test Set（至少 15 个用例）
- [ ] 每个项目接入 Langfuse 可观测
- [ ] 每个项目有 README（含架构图、如何运行、已知问题）
- [ ] 每个项目代码在 GitHub 公开
