# Context Database（上下文数据库）

> 上下文数据库是面向 AI Agent 的新一代数据基础设施，用结构化层次替代传统 RAG 的扁平向量存储。

## 为什么需要上下文数据库

传统 RAG（检索增强生成）在 Agent 场景下暴露出五个系统性缺陷：

1. **上下文碎片化**：记忆、资源、技能分散在向量数据库、文件系统、数据库中，Agent 需要在多个系统间跳转
2. **上下文膨胀**：长任务每次执行都累积上下文，Token 消耗线性增长
3. **检索缺乏全局性**：扁平向量搜索只看语义相似度，不理解上下文之间的层级和组织关系
4. **不可观测**：检索链路隐式运行，出了错难以定位
5. **记忆无法迭代**：只有对话级记忆，缺少任务级经验的沉淀和复用

## 核心设计原则

### 1. 文件系统范式

用 URI 路径（如 `viking://resources/docs/design.md`）组织上下文，而非扁平 ID。这带来三个优势：
- **确定性**：路径是唯一的，增删改查可预测
- **可浏览**：像 `ls`、`cd` 一样浏览上下文
- **权限天然隔离**：目录级权限控制

### 2. 分层加载（Tiered Loading）

不是把所有内容都塞进 prompt，而是按需加载：
- 先看摘要（~100 tokens），判断是否相关
- 需要时再看概览（~2k tokens），做决策
- 最终才加载全文

这类似于 CPU 的多级缓存：L1 快但小，L3 慢但大。上下文数据库把同样的思想用到 Token 管理上。

### 3. 目录递归检索

```
传统 RAG：query → 向量搜索 → top_k 结果（扁平列表）
上下文数据库：query → 定位目录 → 目录内搜索 → 子目录递归 → 聚合
```

目录递归检索结合了"目录定位的全局性"和"语义搜索的精准性"。

### 4. 可观测检索

每一步浏览、定位、递归的轨迹都被记录，形成可追溯的检索链，类似分布式追踪中的 Span。
